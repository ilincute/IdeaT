Usability Test Plan Section

Test Plan for: Application's GI and recipe recommendation component

Persona Type Tested: Primary/Secondary/Negative

Context: office/home
Duration: 20 minutes

Complete the following before starting: Greeting and Consent & User Background Information.

Overall Objectives

We want to test the ability of the user to insert the glycemic index in the Metrics page and then to check the dishes recommended based on the value he inserted in the Recipes page. With this, we want to test the accesibility of the application.

Scenario

The scenario for a primary persona is that he/she is ar the office and has to check his/hers glycemic index befor lunch. He/She uses the device which measures the blood sugar and then comes back to the laptop and accesses the Nutrics application. He/She now has to enter the value obtained and then check the dishes recommended. Use one scenario for each test.
The scenario for a secondary persona is that he/she has a girlfriend/boyfriend wih diabetes and wants to surprise her/him with a nice dinner. For this, our user has to insert the GI of his/hers beloved one and then choose from the recipe recommendations.
The scenario for the negative persona is that once a year, he/she does blood tests just to be sure he/she is healthy. This year though, he/she has the blood sugar a little over the limits. To lower the blood sugar, he/she uses Nutrics for recipe recommendations.

Tasks

The tasks are the same for all the persona types.

  • Task 1: Log into the application with a social media account. ETA: 30 seconds
  • Task 2: Find the page where to insert the GI and check the graphic generated by the value. ETA: 1 minute
  • Task 3: Find the recipes page and open a recipe recommendation from a category of your choice (breakfast/lunch/dinner/snack). ETA: 30 seconds
Post-test Questions

These questions could be presented to the user as a questionnaire or verbally asked.

  • Did you find it difficult to login into the application?
  • Do you think it is necessary to login intro the application or just use it as it is? Why?
  • Did you find something difficult to do among the tasks?
  • Were you pleased with the recommendations received?
Notes for Test Coordinator
  • Remind the user that you would like them to 'think aloud' so that you can understand what they are thinking, about the task or the product/application;
  • Read the user the scenario for the test;
  • Read the user the tasks he has to complete. If the tasks are complex, you may want to put a sheet of paper in front of the user with each task written out;
  • Record the user's actions, comments, questions, and body language (write them in the Test Observations section accompanied by video/audio).
  • Don't offer help; let the user attempt to perform the task. If they ask for help reply with: "What do you think you/that would do?" "What do you think that means?".

Test Results for: Application's GI and recipe recommendation component

Success Criteria

A successful design has been achieved when:

  • 100% of users succeed to login
  • 66% of users succeed to insert the GI
  • 33% of users succeed to find the recipe from the category they want

Summary

List major findings in list or table format for quick and easy reading. E.g. "A number of users (specify the number) did/found _______"

  • One out of three users found it difficult to find the recipe in the Recipes page because he didn't know he had to click on the category first.
  • All of the three users succeeded to login.
  • One out of three users expected to see the recommendations in the same page as the GI.
